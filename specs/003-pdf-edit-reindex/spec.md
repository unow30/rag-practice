# Feature Specification: 업로드 PDF 편집 반영 및 재처리

**Feature Name**: 업로드 PDF 편집 반영 및 재처리  
**Short Name**: pdf-edit-reindex  
**Created**: 2026-04-20  
**Updated**: 2026-04-20  
**Status**: Draft

---

## Overview

사용자가 이미 업로드된 PDF 문서를 외부에서 편집(하이라이트, 밑줄, 취소선, 메모 추가)한 후 해당 변경 사항을 RAG 시스템에 반영할 수 있도록, 문서를 다시 처리(재추출 → 재청킹 → 재임베딩 → 재인덱싱)하는 기능을 제공한다. 또한 PDF 파일 내 네 가지 주석(하이라이트·밑줄·취소선·메모) 정보를 텍스트 추출 시 함께 파싱하여 RAG 검색 및 질의응답에 활용할 수 있도록 한다.

---

## Problem Statement

현재 시스템은 문서 업로드 시 1회만 처리를 수행하므로, 사용자가 PDF에 하이라이트·밑줄·취소선·메모 등의 주석을 추가하더라도 RAG가 이를 인식하지 못한다. 사용자가 직접 남긴 메모나 강조 정보를 기반으로 질문할 때 시스템이 해당 구간을 구별하지 못하여 활용성이 저하된다.

---

## Goals

- PDF 내 하이라이트·밑줄·취소선·메모 주석 정보를 텍스트 추출 시 함께 파악한다.
- 주석 유형별로 구별된 메타데이터를 청크에 포함하여 검색·질의에 활용한다.
- 메모(텍스트 주석)의 내용도 청크 콘텐츠 또는 메타데이터에 포함하여 검색 가능하게 한다.
- 이미 처리된 문서를 사용자 요청에 따라 재처리할 수 있다.
- 재처리 시 기존 문서 ID를 유지하여 연관 데이터(대화 기록 등)가 보존된다.
- 재처리 진행 상태를 사용자에게 실시간으로 표시한다.

---

## Non-Goals

- PDF 내용 자체를 웹 앱 내에서 편집하는 기능
- 메모·하이라이트·밑줄·취소선 이외의 주석 유형 처리 (스탬프, 도형 등)
- 하이라이트 색상별 의미 분류 (모든 색상을 동일하게 처리)
- 자동 감지 후 무조건 재처리 (사용자가 명시적으로 요청해야만 실행)
- 이미지·스캔 PDF에서의 주석 인식 (텍스트 기반 PDF만 지원)

---

## User Scenarios & Testing

### 시나리오 1: 주석이 포함된 PDF 신규 업로드

**Actor**: 일반 사용자  
**전제조건**: 사용자가 하이라이트·밑줄·취소선·메모 주석이 포함된 PDF 파일을 보유하고 있다.

1. 사용자가 주석이 포함된 PDF를 업로드한다.
2. 시스템이 텍스트를 추출하면서 각 주석 유형(하이라이트·밑줄·취소선·메모)과 해당 구간 및 내용을 함께 추출한다.
3. 주석이 포함된 청크에 주석 유형을 나타내는 메타데이터가 부여되고, 메모의 경우 메모 내용도 함께 저장된다.
4. 사용자가 "밑줄 친 내용이 무엇인가요?", "내가 메모한 내용은?" 등으로 질문한다.
5. 시스템이 해당 주석 유형 메타데이터를 활용하여 관련 구절을 우선 반환한다.

**성공 기준**: 네 가지 주석 유형이 각각 올바르게 인식되어 유형별로 구별된 메타데이터로 저장되고, 메모 내용이 검색 가능한 형태로 저장된다.

---

### 시나리오 2: 기존 업로드 문서에 강조 주석 추가 후 재처리

**Actor**: 일반 사용자  
**전제조건**: PDF가 이미 시스템에 업로드·처리되어 있다.

1. 사용자가 외부 PDF 편집기로 문서에 하이라이트·밑줄·취소선을 추가한다.
2. 사용자가 편집된 PDF를 `data/documents/` 폴더에 덮어쓴다.
3. 사용자가 웹 앱의 문서 목록에서 해당 문서의 "재처리" 버튼을 클릭한다.
4. 시스템이 기존 인덱스·청크 데이터를 삭제하고 새로운 파이프라인을 실행한다.
5. 처리 완료 후 문서 상태가 "준비 완료"로 표시된다.
6. 이후 질의에서 강조 주석 정보가 반영된 결과가 반환된다.

**성공 기준**: 재처리 후 기존 문서 ID가 유지되며, 세 가지 강조 유형 정보가 모두 검색에 반영된다.

---

### 시나리오 3: 재처리 중 상태 확인

**Actor**: 일반 사용자  
**전제조건**: 재처리가 진행 중이다.

1. 사용자가 재처리 진행 상태(추출 중 / 청킹 중 / 임베딩 중)를 확인한다.
2. 처리 완료 또는 실패 시 결과 메시지를 확인한다.

**성공 기준**: 처리 단계가 실시간으로 표시되며, 실패 시 원인 메시지가 제공된다.

---

## Functional Requirements

### 주석 파싱

- FR-01: 시스템은 PDF 텍스트 추출 시 하이라이트(Highlight)·밑줄(Underline)·취소선(StrikeOut)·메모(Text/FreeText) 주석을 각각 감지한다.
- FR-02: 하이라이트·밑줄·취소선에 해당하는 텍스트 구간을 일반 텍스트와 함께 추출한다.
- FR-03: 메모 주석의 경우 메모가 달린 위치의 본문 텍스트 구간과 메모 내용(사용자가 작성한 텍스트)을 함께 추출한다.
- FR-04: 주석 유형 정보는 청크 메타데이터에 유형별로 구별되어 저장된다 (예: `annotation_types: ["highlight", "memo"]`).
- FR-05: 메모 내용은 해당 청크의 검색 가능한 필드에 별도 저장되어 "내가 메모한 내용" 형태의 질문에 응답할 수 있다.
- FR-06: 하나의 텍스트 구간에 여러 주석 유형이 중첩된 경우 모든 유형을 함께 기록한다.
- FR-07: 주석이 없는 PDF는 기존 추출 방식과 동일하게 처리된다.

### 재처리 API

- FR-08: 문서 ID를 기반으로 재처리를 요청하는 API 엔드포인트를 제공한다.
- FR-09: 재처리 시 기존 청크 레코드와 벡터 인덱스를 삭제 후 새로 생성한다.
- FR-10: 재처리 중 문서 상태는 기존 처리 상태값(EXTRACTING, CHUNKING, EMBEDDING 등)을 재활용한다.
- FR-11: 재처리가 완료되면 문서 ID 및 연관 메타데이터(이름, 업로드 일시 등)는 변경되지 않는다.
- FR-12: 동일 문서에 대해 재처리가 이미 진행 중일 때 추가 요청은 거부된다.

### 프론트엔드

- FR-13: 문서 목록의 각 문서에 "재처리" 버튼을 제공한다.
- FR-14: 재처리 진행 중에는 버튼이 비활성화되며 진행 상태가 표시된다.
- FR-15: 재처리 완료 또는 실패 시 사용자에게 알림을 표시한다.

---

## Success Criteria

- 주석(하이라이트·밑줄·취소선·메모)이 포함된 PDF 업로드 시, 각 유형별 주석 구간 텍스트의 90% 이상이 올바르게 메타데이터로 기록된다.
- 주석 유형별 질문("밑줄 친 내용", "내가 메모한 내용" 등) 시, 해당 유형 메타데이터를 보유한 청크가 일반 청크보다 우선적으로 반환된다.
- 메모 내용이 검색 가능한 형태로 저장되어, 메모 텍스트를 포함한 질의응답이 가능하다.
- 재처리 완료 후 기존 문서 ID가 유지된다.
- 재처리 요청 후 처리 완료까지의 소요 시간이 신규 업로드 처리 시간과 동일 수준 이내이다.
- 재처리 중 상태가 사용자에게 실시간으로 표시된다.
- 재처리 실패 시 구체적인 오류 메시지가 제공된다.

## Key Entities

| 엔티티 | 변경 사항 |
|--------|-----------|
| `Chunk` | `annotation_types` 필드(문자열 배열, nullable) 추가 — `highlight`, `underline`, `strikeout`, `memo` 값 포함 |
| `Chunk` (메모) | `memo_content` 필드(텍스트, nullable) 추가 — 사용자가 작성한 메모 내용 저장 |
| `Document` | 재처리 API 경로 추가, 상태 재활용 |
| FAISS Index | 재처리 시 기존 인덱스 파일 삭제 후 재생성 |
| BM25 Index | 재처리 시 기존 인덱스 파일 삭제 후 재생성 |

## Assumptions

- 하이라이트·밑줄·취소선·메모는 표준 PDF Annotation 형식(ISO 32000: Highlight/Underline/StrikeOut/Text/FreeText)으로 저장되어 있다고 가정한다.
- 사용자가 편집된 PDF를 직접 서버의 `data/documents/` 경로에 배치하거나, 재업로드 방식으로 교체한다.
- 하이라이트 색상 구별은 이번 범위에 포함하지 않으며, 모든 색상을 동일하게 처리한다.
- 재처리 중 해당 문서에 대한 질의는 이전 인덱스 기준으로 응답하거나, 처리 중 상태 안내를 반환한다.

---

## Dependencies

- PyMuPDF(fitz) 라이브러리의 `page.annots()` API — 하이라이트·밑줄·취소선·메모 주석 접근 (annot.type 값: 0=Text, 2=FreeText, 8=Highlight, 9=Underline, 11=StrikeOut)
- 기존 `process_document()` 파이프라인 — 재처리 시 재활용
- 기존 문서 상태 관리 로직 — 재처리 상태 표시에 재활용
